    怎么我用了 Goroutine 还变慢了？

    怎么我用了 Goroutine 还变慢了？
    December 5, 2023 · 2 min · 230 words · Me

    在 Reddit 看到了个 Goroutine 相关帖子：
    一个 Go 新手实现的逐行读取文件并计算的程序，不用 Goroutine 的执行时间是 1.5 秒，用了 Goroutine 变成 2.5~3 秒了。 为什么用了 Goroutine 时间比不用还慢？

    来看一下具体代码：

    func main() {
    file, err := os.Open("input.txt")
    sum := 0
    wg := &sync.WaitGroup{}
    resultChan := make(chan int, 1000)

    if err != nil {
       fmt.Println("Error opening file")
       return
    }
    defer file.Close()

    scanner := bufio.NewScanner(file)

    now := time.Now()
    fmt.Println(now)

    for scanner.Scan() {
       wg.Add(1)
       line := scanner.Text()
       // fmt.Println(line)
       go calibrate(line, wg, resultChan)
    }
    if err := scanner.Err(); err != nil {
       fmt.Println("Error reading from file:", err)
    }
    wg.Wait()
    close(resultChan)
    for result := range resultChan {
       sum += result
    }
    fmt.Println(sum)
    elapsed := time.Since(now)
    fmt.Println(elapsed)
}

func calibrate(input string, wg *sync.WaitGroup, resultChan chan int) {
    firstNum, lastNumber, finalDigit := -1, -1, -1
    defer wg.Done()
    for _, c := range input {
       if digit, err := strconv.Atoi(string(c)); err == nil {
          if firstNum == -1 {
             firstNum = digit
          }
          lastNumber = digit
       }
    }    if firstNum == -1 {
       resultChan <- 0
       return
    }
    finalDigit = firstNum*10 + lastNumber
    resultChan <- finalDigit
}

    这段代码的目的是逐行读取文件内容，并对每行进行某种计算。理论上，使用 Goroutines 应该能够并行处理每行的计算，从而提高程序的整体效率。但实际结果却是，执行时间从 1.5 秒增加到了 2.5~3 秒。

    为啥呢？

    看到下面一个很生动的例子来说明原因，描述一下大概是：

    想象一下，如果你是一位学校老师，有 1000 个数学题需要完成。每个数学题只需 1 秒钟。但是拿起一叠纸或归还它们需要每个学生 5 秒钟。

    • 给一个学生整叠纸需要 5 秒钟。学生做数学题需要 1000 秒，然后再花 5 秒钟归还纸堆。总共需要 1010 秒。

    • 你认为让 1000 个学生各自拿一张纸会更快。这样分发纸张需要 5000 秒。每个学生用 1 秒钟（但是并行）做数学题。然后你收集所有的纸张，又需要 5000 秒。总共需要 10,001 秒。哎呀，这更糟糕。

    • 现在想象你将 1000 张纸分给 10 个学生，每个学生得到 100 个数学题的一叠。这样分发工作需要 50 秒，做这些题需要 100 秒，然后归还纸堆需要 50 秒。总共时间是 200 秒。

    每个学生代表一个 Goroutine，数学题则是任务。虽然 Goroutines 本身的开销不大（也有几百上千条指令），但这里的任务太轻了，像折一两只纸鹤那样简单，使用过多的 Goroutine 只会增加沟通和协调成本。

    这个 case 其实比较极端，生产环境估计也没人会这么干，看个乐子就好。
